# jido_hive

`jido_hive` is a **collaborative AI participation substrate** where heterogeneous agents and humans coordinate via shared epistemic spaces.

## The Dream: From Workflow to Substrate

`jido_hive` was originally a workflow execution system with collaboration features. It has now evolved into a true **participation substrate with workflow as a pluggable policy**. 

The substrate does not know about "rounds," "turns," or specific collaboration styles. It provides the low-level primitives: a shared coordination container (the **Room**) that accumulates typed knowledge (**Context Objects**), generated by actors (**Participants**) constrained by structured rules (**Contribution Contracts**), and orchestrated by pluggable rulesets (**Dispatch Policies**).

This separation allows you to run round-robin debate, resource pooling, adversarial evaluation, or human-gated reviews all on the exact same underlying infrastructure.

## The Five Core Primitives

The system's architecture is strictly built upon these primitives:

1. **The Room**: A shared epistemic space with a defined participant set, a durable event log, and a lifecycle. It is a coordination container that doesn't care what work happens inside it, only who is in it, what has been produced, and its current state.
2. **The Context Object**: The atom of the substrate. Instead of passing flat prompts, participants exchange typed payloads (e.g., belief, note, question, decision, artifact). Every context object carries full **provenance** (who produced it, from what inputs, with what authority/confidence) and scope.
3. **The Participant**: Any entity that can consume and produce context objects. This includes local LLM workers, remote APIs, deterministic scripts, or **human operators**. Participants declare their capabilities and resource contributions upon joining.
4. **The Contribution Contract**: A strict agreement every participant must fulfill when writing to a room. It defines the allowed contribution types, object types, and relation types. The substrate validates this contract before accepting the contribution into the room state.
5. **The Dispatch Policy**: Pluggable logic running *inside* the room. Given the current room state and available participants, the policy decides who should produce the next context object and what their filtered input view (`context_view`) should be.

## System Architecture

The substrate is physically split into two Elixir applications:

- [`jido_hive_server`](jido_hive_server/README.md): The **Room Authority and Context Store**. It owns the authoritative room state, runs dispatch policies, manages the Phoenix channel relay, persists context objects with full provenance to SQLite, and exposes REST APIs for humans and operators.
- [`jido_hive_client`](jido_hive_client/README.md): The **Participant Runtime**. A local execution boundary that connects to the server. It receives a scoped `context_view`, executes a local assignment against a `contribution_contract`, and returns typed context objects.

Because execution is decoupled from orchestration, you can plug in any backend execution engine on the client without modifying the server's coordination logic.

## Fastest Local Onboarding

From a fresh clone:

```bash
git clone <repo>
cd jido_hive
bin/setup
```

Open three terminals.

Terminal 1 (Server):
```bash
bin/live-demo-server
```

Terminal 2 (Worker 1):
```bash
bin/client-worker --worker-index 1
```

Terminal 3 (Worker 2):
```bash
bin/client-worker --worker-index 2
```

What you will see:
- The server starts, running its `Ecto` migrations.
- Workers connect over the Phoenix WebSocket relay, issuing `relay.hello` and `participant.upsert` with their capabilities.
- The live demo creates a Room and selects a Dispatch Policy (e.g., `round_robin/v2`).
- The server opens an **Assignment** based on the policy and sends it to the chosen worker.
- The worker executes locally and responds with a structured **Contribution**.
- The server's pure `EventReducer` reduces the contribution, extracting Context Objects into the authoritative room state.

## Recommended Operator Flow

Use the menu-driven bash tools to orchestrate the substrate manually:

Terminal 1:
```bash
bin/hive-control
```

Terminal 2:
```bash
bin/hive-clients
```

1. In `bin/hive-clients`, start two or more workers.
2. In `bin/hive-control`, create a room and run it. Inspect target capabilities, room state, timeline projections, and publications.

Production equivalents:
```bash
bin/hive-control --prod
bin/hive-clients --prod
```

## Useful Local Commands

Repo-level tools:
```bash
bin/setup
bin/server
bin/live-demo-server
bin/hive-control
bin/hive-clients
bin/client-worker --worker-index 1
setup/hive help
```

Orchestrating rooms manually via the CLI:
```bash
setup/hive wait-server
setup/hive wait-targets --count 2
setup/hive create-room room-manual-1 --participant-count 2
setup/hive run-room room-manual-1 --turn-timeout-ms 180000
setup/hive publication-plan room-manual-1
setup/hive publication-runs room-manual-1
```

## Production Smoke Test

1. Tail prod server logs (using Coolify):
```bash
cd jido_hive_server
MIX_ENV=coolify mix coolify.app_logs --project server --lines 200 --follow
```

2. Run prod workers:
```bash
cd /home/home/p/g/n/jido_hive
bin/client-worker --prod --worker-index 1
bin/client-worker --prod --worker-index 2
```

3. Wait for the workers to register capabilities:
```bash
setup/hive --prod wait-targets --count 2
```

4. Run the end-to-end production flow:
```bash
setup/hive --prod live-demo --participant-count 2
```

## Development and Quality

The system is built with a **Functional Core, Imperative Shell** philosophy. Reducers and policies are pure functions; boundary modules adapt to Phoenix and Ecto. Follow TDD for all core extensions.

Repo-wide quality gate:
```bash
mix ci
```

Useful monorepo tasks:
